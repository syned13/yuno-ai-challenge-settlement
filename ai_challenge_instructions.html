Domain Background: Payment Settlement & Reconciliation

To understand this challenge, you need to know how money flows in digital payments:

The Payment Lifecycle

When a customer makes a purchase:

  1. Authorization: The merchant requests permission to charge the customer's card or payment method. The processor (the company that handles the transaction) sends this request to the customer's bank. If approved, the funds are "reserved" but not yet transferred. The merchant receives an authorization code and an authorization amount.

  2. Capture: The merchant confirms they want to collect the funds (usually when they ship the product). Sometimes the capture amount is different from the authorization amount (e.g., final total adjusted for tips, or partial capture of a reservation).

  3. Settlement: This is when money actually moves. The processor batches captured transactions and transfers the funds to the merchant's bank account, typically 1-7 days after capture. The processor sends a settlement file (often CSV, JSON, or via API) listing all transactions settled in that batch, the actual amounts transferred, and any fees deducted.

Why Reconciliation Matters

There are many reasons settlement might not match authorization:

  • Partial captures: A hotel authorizes $500 but only captures $350 after checkout
  • Fees and deductions: Processors deduct transaction fees, interchange fees, or chargeback fees from settlement
  • Disputes and chargebacks: A customer disputes a charge; the processor reverses the settlement
  • Failed settlements: A transaction was authorized but the capture failed, so it never settled
  • Currency conversion: Authorization in BRL but settlement in USD with exchange rate applied
  • Delays: Transactions authorized on one day but settled days or weeks later
  • Duplicate or missing records: Data quality issues in either system

Reconciliation is the process of matching each settled transaction back to its original authorization, identifying discrepancies, and categorizing them so the finance team knows what action to take (e.g., investigate, write off, retry, dispute with processor).

Key Terms

  • Settlement file: A data file from a payment processor listing all transactions that were settled (paid out) in a given period
  • Transaction ID / Reference: A unique identifier for a payment (may differ between internal systems and processor systems)
  • Net settlement amount: The amount actually deposited to the merchant after all fees are deducted
  • Gross amount: The original transaction amount before fees
  • Reconciliation status: Whether a transaction has been matched (reconciled), is unmatched, or has a discrepancy

Your Mission

Build a backend service that performs automated settlement reconciliation for AuraCommerce.

Functional Requirements

1. Transaction Matching Engine

Your service must accept two data sources as input:

  • Internal transaction records: AuraCommerce's database of payment authorizations/captures
  • Settlement reports: Files or data from payment processors showing what actually settled

The service should:

  • Match settlement records to internal transaction records based on identifiers (transaction ID, order ID, amount, date, etc.)
  • Handle cases where identifiers don't match perfectly (e.g., processor uses a different reference format)
  • Categorize each transaction into one of these reconciliation states:
    • Matched: Settlement found and amounts align (within acceptable tolerance for fees/FX)
    • Matched with variance: Settlement found but amount differs from expected (identify the difference)
    • Unsettled: Internal record exists but no corresponding settlement found
    • Unexpected settlement: Settlement record exists but no corresponding internal transaction found
    • Duplicate: Multiple settlement records for the same transaction

2. Discrepancy Analysis & Reporting

The service must produce a reconciliation report that includes:

  • Summary statistics: total matched, total unmatched, total variance amount, reconciliation rate
  • Breakdown by currency, country, and processor
  • Detailed list of all discrepancies with sufficient context to investigate (transaction details, expected vs actual amounts, date ranges, reference IDs)
  • Identification of high-priority discrepancies (e.g., large amounts, old unsettled transactions)

The report format is up to you (JSON, CSV, PDF, HTML, etc.) but it must be machine-readable and human-friendly.

3. API Interface

Expose an API (REST or GraphQL) that allows:

  • Uploading/submitting transaction data and settlement data
  • Triggering reconciliation runs
  • Retrieving reconciliation reports and status
  • Querying specific transaction reconciliation status

Stretch Goals (Partial completion expected and welcomed)

  • Multi-currency reconciliation: Handle foreign exchange conversions when authorization currency differs from settlement currency (use static or approximate FX rates for demo purposes)
  • Configurable matching rules: Allow configuration of matching tolerance (e.g., "amounts within 2% are considered matched" to account for FX fluctuations or rounding)
  • Time-window analysis: Flag transactions that settled outside expected time windows (e.g., settlements more than 7 days after authorization)
  • Historical trend detection: Identify processors or transaction types with chronic reconciliation issues
